# **V0.2 Architecture Proposal: Grip & Glial Session Persistence and Synchronization**

## **1\. Executive Summary**

This document outlines the design for incrementally persisting, restoring, and synchronizing the Context Graph in the Grip engine. It addresses the challenges of dynamic graph topology (where taps and contexts are swapped in and out via matchers), headless client restoration, and provides a foundation for full active synchronization (Project Glial) and AI Assistant integration.

## **2\. Stable Addressing & Identifiers**

To persist state independent of the active memory graph, every piece of state-bearing infrastructure needs a deterministic, reproducible address.

### **2.1 Drip Identifiers**

Drips (instances of Grips) already have a natural hierarchical identity.

* **Format:** \[Context Path\]\#\[Grip ID\]  
* **Example:** /root/session/user(123)\#preferences.theme  
* **Storage payload:** Simply the primitive value or serialized JSON object.

### **2.2 Tap Identifiers**

Taps are compute nodes. While pure functional taps hold no state, *Async* and *Effect* taps hold internal state (e.g., caches, debounce timers, dynamic destination keys).

* **Implicit ID (Hash):** If a tap is not explicitly named, its ID is deterministically generated by hashing its signature: hash(sorted\_input\_grips \+ sorted\_output\_grips).  
* **Explicit ID:** Developers can pass a name to the tap configuration.  
* **Format:** \[Context Path\]@tap:\[Tap ID\]  
* **Storage payload:** A Tap State Object.  
  * *Atom Taps:* { lastValue: "..." }  
  * *Async Taps:* { cache: { reqKey: resValue }, destContextKey: "/root/dynamic(456)" }

## **3\. The "Hydration Buffer" (Solving Partial Initialization)**

Because the context graph is dynamic (matchers swap topologies based on state), we cannot build the entire graph on startup. Instead, we introduce the **Hydration Buffer**.

### **3.1 Mechanism**

1. **Boot Phase:** On engine start, the Storage Adapter loads all persisted state into an in-memory HydrationBuffer (a flat key-value map).  
2. **Lazy Injection:** When the Engine creates a new Context, Drip, or Tap (either at startup or later via a Matcher swapping in a new sub-graph), the entity queries the HydrationBuffer using its Stable ID.  
3. **Consumption:**  
   * If a Drip finds its ID in the buffer, it initializes with that value instead of its default.  
   * If a Tap finds its ID, it calls its internal state\_put(hydrated\_state) to restore caches or dynamic destinations.  
4. **Cleanup:** Once a context is permanently destroyed (and not just temporarily hidden), the engine can issue a DELETE to the storage adapter to garbage collect the orphaned state.

## **4\. Incremental State Storage API**

## **4\. Incremental State Storage API**

To support seamless incremental updates without full-graph serialization, we use an event-driven Storage Adapter interface. The Engine emits granular deltas.

interface StateDelta {  
  id: string;          // Stable ID (Drip or Tap)  
  type: 'drip' | 'tap';  
  value: any;          // Full replacement value (for primitives) OR a Patch object  
  patch?: JsonPatch\[\]; // RFC 6902 array of operations for complex objects  
  hlc: string;         // Hybrid Logical Clock timestamp (e.g., "1690800000000-0000-clientA")  
  causalityId: string; // ID of the root action/event that triggered this  
}

interface StorageAdapter {  
  // Stage 1: Session Storage  
  loadAll(): Promise\<Record\<string, any\>\>; // Feeds the Hydration Buffer  
  put(delta: StateDelta): void;            // Called by Engine on change  
  remove(id: string): void;                // Called on context destruction  
    
  // Stage 2: Active Synchronization (Glial)  
  onRemoteDelta(callback: (delta: StateDelta) \=\> void): void;  
}

* **Drip State put:** Fired immediately when a Drip value changes.  
  * **Tap State put:** Fired by the Tap itself when its internal state changes (e.g., an async request completes and the cache updates).

## **5\. Glial: Two-Stage Synchronization**

Glial extends the StorageAdapter to connect distributed graphs.

### **Stage 1: Session Storage (Local/Remote KV)**

* **Implementation:** LocalStorageAdapter or RedisSessionAdapter.  
* **Behavior:** put() serializes the delta and writes it to the store. loadAll() happens once on startup. No active listening.

### **Stage 2: Active Synchronization (Glial Relay)**

* **Implementation:** GlialWebsocketAdapter.  
* **Behavior:**  
  * put() sends the StateDelta over a WebSocket to the Glial Server.  
  * onRemoteDelta() listens for incoming WebSockets.  
* **Loop Prevention (CRITICAL):** When a delta arrives from Glial, the Engine applies it to the Drip. The Drip will emit a change event. The adapter must use a "taint" flag (e.g., origin: 'remote') to ensure it does not bounce the delta back to the server.

## **6\. Headless Client Graph Restoration**

Headless clients (e.g., a Python backend providing LLM taps or DB hooks) present a unique challenge: they don't have user local storage, and they may restart unexpectedly.

1. **Connection Handshake:** When the headless client connects to the Glial Server, it identifies the session it is managing.  
2. **State Plane Dump:** The Glial Server sends the complete current state of that session (the Hydration Buffer).  
3. **Graph Bootstrapping:** The headless engine initializes its Root context. As its internal matchers evaluate the hydrated state, they rapidly build out the necessary sub-contexts and attach the required infra-taps.  
4. **Resuming Async:** Any Async taps that were mid-flight during the crash will notice they lack a cached result for the current inputs and will automatically re-fire.

## **7\. AI Assistant State Semantics**

An AI assistant needs to understand *what* the graph state means, not just read raw JSON. We enable this by extending Grip definitions with a Semantics layer.

### **7.1 Semantic Annotations**

When defining a Grip, developers provide AI-facing metadata:

const UserLocationGrip \= defineGrip({  
  id: "user.location",  
  type: "string",  
  semantics: {  
    description: "The city the user is currently looking at for weather data.",  
    mutable: true, // Can the AI change this?  
    action\_intent: "change\_location",  
  }  
});

### **7.2 The "Semantic Snapshot"**

We provide a utility: engine.exportSemanticState(). This traverses the currently active contexts and outputs an LLM-optimized JSON schema, enabling an AI to inject commands deterministically.

## **8\. Distributed Concurrency: A Local-First Approach**

Requiring a round-trip to the Glial Server to validate a version sequence is unacceptable for highly interactive UIs. The system must be **Local-First**, allowing the local Grip engine to commit state immediately and synchronize asynchronously without blocking.

To achieve this, we replace central versioning with a decentralized, CRDT-inspired (Conflict-Free Replicated Data Type) resolution mechanism.

### **8.1 Hybrid Logical Clocks (HLC)**

Instead of a server-owned version number, every client (Browser, AI, Headless Backend) generates its own timestamps using a **Hybrid Logical Clock (HLC)**.

An HLC guarantees a globally unique, sortable timestamp for every event, even if system clocks drift.

* **Format:** \<physical\_time\>-\<logical\_counter\>-\<client\_id\>  
* **Example:** 1715000000000-0001-browser\_abc123  
* **Behavior:** When a client receives a delta from Glial, it advances its own local HLC physical time to match the incoming time (if the incoming time is strictly greater than local time). This prevents a client with a "fast" clock from permanently overriding everyone else.

### **8.2 Field-Level Last-Writer-Wins (LWW)**

Because there is no central sequencer enforcing order, deltas *will* arrive out of order across distributed nodes. We resolve this mathematically using the HLC.

Every stored property in the Hydration Buffer (and memory) must secretly retain the HLC of its last mutation.

1. **Local Mutation:** A user types in an input. The local Drip updates immediately, generates an HLC, and sends the StateDelta to the Glial server. UI is instantly responsive.  
2. **Receiving a Delta:** When a client (or the server) receives a delta, it compares the incoming hlc with the local hlc for that specific Drip.  
3. **The LWW Rule:** \* If incoming\_hlc \> local\_hlc: Apply the change.  
   * If incoming\_hlc \< local\_hlc: Ignore the incoming delta entirely (it's a stale, delayed packet from the past).

### **8.3 The Delta Protocol (Conflict-Free Patching)**

For large JSON objects, sending full payloads is too heavy, but standard JSON Patch (RFC 6902\) breaks if applied out of order.

To make patches commutative (order-independent), we augment our patching logic to act as an **LWW-Map** (a basic CRDT):

* **Granular Patch HLCs:** Instead of the whole JSON object having one timestamp, the delta payload maps changes to specific paths, and the local engine tracks the HLC *per property path*.  
* **Example:** Client A updates /metadata/title at T1. Client B updates /metadata/author at T2. Both send patches. Because the paths do not intersect, both apply cleanly regardless of which arrives first.  
* **Collision:** If Client A and Client B both update /metadata/title, the patch with the highest HLC wins, and the loser is silently ignored.

This guarantees **Eventual Consistency** across all nodes (Browser, Headless Backend, AI) with zero network blocking on the critical path of the UI.
